## 一、本课目标

学完本课，你应该能做到：

1. 理解为什么业务系统需要结构化输出，而不能长期依赖自由文本。
2. 理解 LangChain 为什么设计 `with_structured_output`。
3. 使用 Pydantic 定义模型输出结构。
4. 使用 `with_structured_output()` 让模型返回程序可消费的数据对象。
5. 实现一个招聘信息提取器雏形。

前三课我们已经学会：

- 调用聊天模型。
- 用 Messages 和 Prompt 模板组织输入。
- 查看 `AIMessage`，并用 `StrOutputParser` 取出文本。

这一课要迈出非常关键的一步：不再只让模型“写一段话”，而是让模型“按字段返回数据”。

---

## 二、为什么会有结构化输出

先看一个真实需求。

你要做一个“招聘信息提取器”，输入一段自然语言招聘描述，程序要提取：

- 职位名称。
- 公司名称。
- 工作地点。
- 技能要求。
- 薪资范围。
- 是否远程。

如果让模型自由回答，可能得到：

```text
这个岗位是后端工程师，公司是星河科技，地点在上海，需要 Python、FastAPI 和 PostgreSQL，薪资大约 25k-35k，不支持远程。
```

这段话人能看懂，但程序不好用。

程序如果想把它存进数据库，需要知道：

```text
job_title = ?
company = ?
location = ?
skills = ?
salary = ?
remote = ?
```

如果继续用第 3 课的自由文本处理，就可能要写：

- 正则表达式。
- 字符串切分。
- 关键词查找。
- 各种格式兜底。

问题是，模型下一次可能换一种说法：

```text
星河科技正在上海招聘后端开发，要求熟悉 Python/FastAPI/PostgreSQL。待遇 25-35K，需到岗办公。
```

意思一样，格式变了。

这就是结构化输出存在的原因：

> LLM 生成的自由文本适合给人读；结构化输出适合给程序用。

LangChain 设计结构化输出，是为了把“让模型按固定字段回答、把结果校验成程序对象”这件事变成统一能力，而不是让每个开发者都手写解析逻辑。

---

## 三、结构化输出解决了什么复杂度

结构化输出把下面几类复杂度封装起来：

| 复杂度 | 没有结构化输出时 | 使用结构化输出后 |
| --- | --- | --- |
| 输出格式 | 依赖模型自觉遵守 Prompt | 由 schema 明确字段 |
| 字段含义 | 靠自然语言描述 | 每个字段有类型和说明 |
| 程序消费 | 手动解析字符串 | 直接拿对象属性 |
| 类型检查 | 自己判断字符串、数字、列表 | Pydantic 自动校验 |
| 缺失处理 | 靠规则兜底 | 用可选字段表达 |

它的设计意图可以这样理解：

```text
Prompt 负责告诉模型任务是什么。
Schema 负责告诉模型和程序结果应该长什么样。
Pydantic 负责把结果校验成 Python 对象。
```

这让模型输出从“看起来像数据”变成“真的可以被程序使用的数据”。

---

## 四、Pydantic 是什么

Pydantic 是 Python 中常用的数据建模和校验库。

在结构化输出里，它负责描述：

- 有哪些字段。
- 字段是什么类型。
- 字段是什么意思。
- 哪些字段可以为空。
- 模型返回后如何校验。

例如：

```python
from pydantic import BaseModel, Field


class Person(BaseModel):
    name: str = Field(description="姓名")
    age: int = Field(description="年龄")
```

这里的 `Person` 不是给模型看的普通文字，而是一个结构定义。

你可以把它理解为：

```text
模型输出必须尽量符合这个数据结构。
程序最终希望拿到这个 Python 对象。
```

如果你的环境里没有 Pydantic，可以安装：

```powershell
uv add pydantic
```

通常安装 LangChain 相关依赖时已经会带上 Pydantic，但教材中显式说明它，是为了让学习者知道这个库本身的职责。

## 五、示例 1：最小结构化输出

新建文件：

```text
examples/lesson_04_minimal_structured_output.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field


load_dotenv()


class ConceptExplanation(BaseModel):
    """A structured explanation of a technical concept."""

    concept: str = Field(description="The technical concept being explained")
    definition: str = Field(description="A one-sentence definition")
    example: str = Field(description="A simple example for beginners")


model = ChatOpenAI(
    model=getenv("OPENAI_MODEL", "gpt-5-nano"),
    temperature=0.2,
)

structured_model = model.with_structured_output(
    ConceptExplanation,
    method="json_schema",
)

result = structured_model.invoke("请解释 LangChain 中的 PromptTemplate。")

print(type(result))
print(result)
print("\n概念：", result.concept)
print("定义：", result.definition)
print("例子：", result.example)
```

运行：

```powershell
uv run examples/lesson_04_minimal_structured_output.py
```

观察重点：

- `result` 不再是 `AIMessage`。
- `result` 是 `ConceptExplanation` 类型的 Pydantic 对象。
- 可以用 `result.concept`、`result.definition`、`result.example` 直接取字段。

这就是结构化输出和自由文本输出最大的区别：

```text
自由文本：拿到一段话，再想办法解析。
结构化输出：直接拿到一个字段明确的对象。
```

## 六、理解 `with_structured_output`

`with_structured_output()` 是模型对象上的方法。

它的核心作用是：

> 基于你提供的 schema，返回一个新的模型调用对象。这个新对象会尽量让模型按 schema 输出，并把结果解析成对应结构。

代码里这一步非常关键：

```python
structured_model = model.with_structured_output(
    ConceptExplanation,
    method="json_schema",
)
```

这里发生了三件事：

1. `ConceptExplanation` 告诉模型应该输出哪些字段。
2. 字段类型和 `Field(description=...)` 帮助模型理解每个字段含义。
3. LangChain 会把模型返回结果解析成 `ConceptExplanation` 对象。

这里的 `method="json_schema"` 表示使用 OpenAI 原生 structured output 能力。根据官方 OpenAI 集成文档，如果不指定该参数，`ChatOpenAI` 默认会使用 function calling 方式实现结构化输出。

如果你使用的是第三方 OpenAI 兼容接口，而它不支持原生 JSON Schema 结构化输出，可以尝试去掉 `method="json_schema"`，让 LangChain 使用默认方式，或换用该服务商明确支持的模型与参数。

它的设计很像给模型加了一层“输出接口”：

```text
普通模型：输入文本 → AIMessage
结构化模型：输入文本 → Pydantic 对象
```

所以它适合这类任务：

- 信息提取。
- 分类结果。
- 打标签。
- 评分。
- 路由决策。
- 表单填写。

不适合这类任务：

- 长篇创作。
- 开放式闲聊。
- 需要保留丰富自然语言表达的回答。

## 七、示例 2：默认 function calling 方式

前面的示例使用了：

```python
method="json_schema"
```

这适合支持 OpenAI 原生 structured output 的模型和接口。

但在真实学习和开发中，你很可能使用第三方 OpenAI 兼容接口。很多兼容接口支持普通聊天和工具调用，但不一定支持 OpenAI 原生 JSON Schema 结构化输出。

这时可以先尝试默认 function calling 方式，也就是不传 `method` 参数：

```python
structured_model = model.with_structured_output(ConceptExplanation)
```

它的设计意义是：同样使用 schema 描述输出结构，但底层通过工具/函数调用协议来帮助模型返回结构化结果。

新建文件：

```text
examples/lesson_04_function_calling_structured_output.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field


load_dotenv()


class ConceptExplanation(BaseModel):
    """A structured explanation of a technical concept."""

    concept: str = Field(description="The technical concept being explained")
    definition: str = Field(description="A one-sentence definition")
    example: str = Field(description="A simple example for beginners")


model = ChatOpenAI(
    model=getenv("OPENAI_MODEL", "gpt-5-nano"),
    temperature=0.2,
)

# Default method: function calling.
# This is often more compatible with third-party OpenAI-compatible endpoints.
structured_model = model.with_structured_output(ConceptExplanation)

result = structured_model.invoke("请解释 LangChain 中的结构化输出。")

print(type(result))
print(result)
print("\n概念：", result.concept)
print("定义：", result.definition)
print("例子：", result.example)
```

运行：

```powershell
uv run examples/lesson_04_function_calling_structured_output.py
```

观察重点：

- 代码没有传 `method="json_schema"`。
- 返回结果仍然是 `ConceptExplanation` 对象。
- 如果你的兼容接口不支持 `json_schema`，这个版本更可能跑通。
- 如果这个版本也报错，说明当前模型或接口可能连工具调用/function calling 兼容性也不完整。

你可以先记住这个选择：

| 写法 | 更适合什么时候 |
| --- | --- |
| `with_structured_output(Schema, method="json_schema")` | 使用支持原生 structured output 的 OpenAI 模型或兼容接口 |
| `with_structured_output(Schema)` | 需要默认 function calling 方式，或第三方兼容接口不支持 `json_schema` |

## 八、示例 3：招聘信息提取器

现在回到本课核心案例：从招聘描述中提取字段。

新建文件：

```text
examples/lesson_04_job_extractor.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field


load_dotenv()


class JobPosting(BaseModel):
    """Structured information extracted from a job posting."""

    job_title: str = Field(description="The job title, such as Backend Engineer")
    company: str | None = Field(default=None, description="The company name if mentioned")
    location: str | None = Field(default=None, description="The work location if mentioned")
    skills: list[str] = Field(description="Required or preferred skills")
    salary_range: str | None = Field(
        default=None,
        description="The salary range if mentioned",
    )
    remote: bool | None = Field(
        default=None,
        description="Whether the job supports remote work",
    )


model = ChatOpenAI(
    model=getenv("OPENAI_MODEL", "gpt-5-nano"),
    temperature=0.2,
)

structured_model = model.with_structured_output(
    JobPosting,
    method="json_schema",
)

job_text = """
星河科技正在招聘后端工程师，工作地点上海。
希望候选人熟悉 Python、FastAPI、PostgreSQL，有 LangChain 或 RAG 项目经验更好。
薪资范围 25k-35k，需要每周至少 3 天到办公室办公，暂不支持全远程。
"""

result = structured_model.invoke(
    f"请从下面招聘信息中提取结构化字段：\n\n{job_text}"
)

print(result)
print("\n职位：", result.job_title)
print("公司：", result.company)
print("地点：", result.location)
print("技能：", result.skills)
print("薪资：", result.salary_range)
print("是否远程：", result.remote)
```

运行：

```powershell
uv run examples/lesson_04_job_extractor.py
```

观察重点：

- `skills` 应该是 Python 列表，而不是一段逗号分隔字符串。
- `remote` 应该是布尔值或 `None`。
- 没提到的字段可以用 `None` 表示。
- 程序可以直接访问字段，不需要再切分文本。

## 九、字段设计：类型比提示词更重要

结构化输出里，字段设计会明显影响结果质量。

看这个字段：

```python
skills: list[str] = Field(description="Required or preferred skills")
```

它同时告诉模型和程序：

- 这里应该是多个技能。
- 每个技能是字符串。
- 最终程序希望拿到列表。

再看这个字段：

```python
remote: bool | None = Field(
    default=None,
    description="Whether the job supports remote work",
)
```

它表达了三个可能：

- `True`：支持远程。
- `False`：不支持远程。
- `None`：文本没有提到。

这比让模型输出“支持 / 不支持 / 未提及”更适合程序判断。

设计 schema 时要记住：

```text
字段不是越多越好。
字段越清楚，模型越容易稳定输出。
```

好的字段设计通常具备：

- 字段名清晰。
- 类型明确。
- 描述具体。
- 可选字段用 `类型 | None` 表示允许为空，并配合 `default=None` 表示字段可以缺省。
- 列表字段说明元素是什么。

## 十、示例 4：处理缺失字段

自然语言文本经常不完整。

如果招聘描述没有公司名或薪资，结构化输出不应该强行编造。

新建文件：

```text
examples/lesson_04_missing_fields.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field


load_dotenv()


class JobPosting(BaseModel):
    """Structured information extracted from a job posting."""

    job_title: str = Field(description="The job title")
    company: str | None = Field(default=None, description="The company name if mentioned")
    location: str | None = Field(default=None, description="The work location if mentioned")
    skills: list[str] = Field(description="Required or preferred skills")
    salary_range: str | None = Field(
        default=None,
        description="The salary range if mentioned",
    )
    remote: bool | None = Field(
        default=None,
        description="Whether the job supports remote work",
    )


model = ChatOpenAI(
    model=getenv("OPENAI_MODEL", "gpt-5-nano"),
    temperature=0.2,
)

structured_model = model.with_structured_output(
    JobPosting,
    method="json_schema",
)

job_text = """
招聘机器学习工程师，要求熟悉 Python、PyTorch 和向量检索。
工作地点北京，可接受每周两天远程。
"""

```

运行：

```powershell
uv run examples/lesson_04_missing_fields.py
```

观察重点：

- 文本没提到公司，`company` 应该是 `None`。
- 文本没提到薪资，`salary_range` 应该是 `None`。
- Prompt 中明确要求“不要编造未提及的信息”，能帮助模型更稳。

结构化输出不是让模型变成数据库，它仍然需要良好的任务说明和字段描述。

## 十一、结构化输出和 JSON 的区别

很多人第一次学结构化输出会问：

> 我直接让模型返回 JSON 不就行了吗？

例如：

```text
请严格返回 JSON，字段包括 job_title、company、skills。
```

这当然能做一些简单任务，但长期会遇到问题：

- 模型可能多输出解释文字。
- JSON 可能格式不合法。
- 字段类型可能不稳定。
- 你还要自己写解析和错误处理。
- 字段含义散落在 Prompt 里，不如 schema 清晰。

`with_structured_output()` 的价值不只是“让模型像 JSON 一样输出”，而是：

```text
用 schema 统一描述期望输出，并把模型结果解析成程序对象。
```

所以你要区分：

| 做法 | 适用场景 |
| --- | --- |
| 让模型自由输出文本 | 给人阅读 |
| 让模型按 JSON 格式输出 | 小脚本、临时任务 |
| `with_structured_output(PydanticModel)` | 程序长期消费、字段稳定、需要校验 |

### 在 Prompt 模板中写 JSON 示例要使用双花括号

虽然本课推荐优先使用 `with_structured_output()`，但你仍然会在一些场景中需要在提示词里写 JSON 示例，例如：

- 兼容接口暂时不支持结构化输出。
- 你只是做一个很小的临时脚本。
- 你想在 Prompt 中展示目标格式，帮助模型理解字段。

这时要注意一个细节：`ChatPromptTemplate` 和 `PromptTemplate` 会把 `{name}` 识别成模板变量。

如果你在模板中直接写 JSON：

```python
from langchain_core.prompts import ChatPromptTemplate


prompt = ChatPromptTemplate.from_messages(
    [
        (
            "human",
            """
请提取招聘信息，并返回 JSON：

{
  "job_title": "...",
  "company": "...",
  "skills": ["..."]
}

招聘信息：
{job_text}
""",
        )
    ]
)
```

LangChain 可能会把 JSON 里的 `{` 和 `}` 当成模板变量边界，导致模板解析错误。

正确写法是：如果你希望花括号作为普通 JSON 字符出现在 Prompt 中，需要写成双花括号：

```python
from langchain_core.prompts import ChatPromptTemplate


prompt = ChatPromptTemplate.from_messages(
    [
        (
            "human",
            """
请提取招聘信息，并返回 JSON：

{{
  "job_title": "...",
  "company": "...",
  "skills": ["..."]
}}

招聘信息：
{job_text}
""",
        )
    ]
)
```

这里有两类花括号：

| 写法 | 含义 |
| --- | --- |
| `{job_text}` | 模板变量，运行时会被替换 |
| `{{` 和 `}}` | 普通花括号，渲染后会变成 JSON 的 `{` 和 `}` |

也就是说：

```text
模板中的 {{  → 最终 Prompt 中的 {
模板中的 }}  → 最终 Prompt 中的 }
模板中的 {job_text} → 运行时传入的招聘文本
```

你可以把它理解成：单花括号留给 Prompt 变量，双花括号用来写字面量 JSON。

这个规则不只适用于 JSON，也适用于任何你想在模板中保留普通 `{}` 的场景。

## 十二、示例 5：JSON 格式提示词 + `with_structured_output`

现在把这两个能力合在一起：

```text
Prompt 中给出 JSON 格式示例
  +
with_structured_output() 使用 Pydantic schema 解析结果
```

这样做的意义是：

- Prompt 里的 JSON 示例帮助模型理解你希望看到的字段形状。
- `with_structured_output()` 负责把最终结果解析成 Pydantic 对象。
- 程序最终不要解析 Prompt 中的 JSON 文本，而是直接使用结构化对象。

新建文件：

```text
examples/lesson_04_prompt_json_and_structured_output.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field


load_dotenv()


class JobPosting(BaseModel):
    """Structured information extracted from a job posting."""

    job_title: str = Field(description="The job title")
    company: str | None = Field(default=None, description="The company name if mentioned")
    location: str | None = Field(default=None, description="The work location if mentioned")
    skills: list[str] = Field(description="Required or preferred skills")
    remote: bool | None = Field(
        default=None,
        description="Whether the job supports remote work",
    )


prompt = ChatPromptTemplate.from_messages(
    [
        (
            "system",
            "你是一个招聘信息提取助手。请只根据用户提供的文本提取信息，不要编造。",
        ),
        (
            "human",
            """
请从下面招聘信息中提取字段。

你需要理解的目标 JSON 形状如下：

{{
  "job_title": "职位名称",
  "company": "公司名称或 null",
  "location": "工作地点或 null",
  "skills": ["技能1", "技能2"],
  "remote": true
}}

注意：
- 如果字段未提及，请返回 null
- skills 必须是字符串列表
- remote 必须是 true、false 或 null

招聘信息：
{job_text}
""",
        ),
    ]
)

model = ChatOpenAI(
    model=getenv("OPENAI_MODEL", "gpt-5-nano"),
    temperature=0.2,
)

# 默认 function calling 方式，更适合不少第三方 OpenAI 兼容接口。
# 如果你的模型支持原生 structured output，也可以改为：
# structured_model = model.with_structured_output(JobPosting, method="json_schema")
structured_model = model.with_structured_output(JobPosting)

job_text = """
云舟智能招聘 AI 应用工程师，工作地点深圳。
要求熟悉 Python、LangChain、RAG 和 FastAPI。
可每周远程一天。
"""

prompt_value = prompt.invoke({"job_text": job_text})
result = structured_model.invoke(prompt_value)

print(type(result))
print(result)
print("\n职位：", result.job_title)
print("公司：", result.company)
print("地点：", result.location)
print("技能：", result.skills)
print("是否远程：", result.remote)
```

运行：

```powershell
uv run examples/lesson_04_prompt_json_and_structured_output.py
```

观察重点：

- Prompt 中 JSON 示例使用了 `{{` 和 `}}`。
- `{job_text}` 仍然是正常模板变量。
- `prompt.invoke()` 输出的是 PromptValue。
- `structured_model.invoke(prompt_value)` 输出的是 `JobPosting` 对象。
- 代码没有手动解析 JSON 字符串。

这个例子体现了两层约束：

| 层次 | 作用 |
| --- | --- |
| Prompt 中的 JSON 示例 | 帮助模型理解目标字段和格式 |
| Pydantic schema | 让 LangChain 解析并校验为程序对象 |

如果你已经使用 `with_structured_output()`，Prompt 中的 JSON 示例不是必需的；但在字段较多、兼容接口能力不稳定，或者你希望进一步强调输出形状时，它可以作为辅助说明。

## 十三、实践案例：招聘信息提取器增强版

现在你来做一个稍微完整一点的实践案例：用户输入招聘描述，程序输出结构化对象，并给出简单后处理。

新建文件：

```text
examples/lesson_04_job_extractor_cli.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field


load_dotenv()


class JobPosting(BaseModel):
    """Structured information extracted from a job posting."""

    job_title: str = Field(description="The job title")
    company: str | None = Field(default=None, description="The company name if mentioned")
    location: str | None = Field(default=None, description="The work location if mentioned")
    skills: list[str] = Field(description="Required or preferred skills")
    salary_range: str | None = Field(
        default=None,
        description="The salary range if mentioned",
    )
    remote: bool | None = Field(
        default=None,
        description="Whether the job supports remote work",
    )
    seniority: str | None = Field(
        default=None,
        description="The seniority level, such as junior, middle, senior, expert"
    )


model = ChatOpenAI(
    model=getenv("OPENAI_MODEL", "gpt-5-nano"),
    temperature=0.2,
)

structured_model = model.with_structured_output(
    JobPosting,
    method="json_schema",
)

job_text = input("请输入招聘描述：")

result = structured_model.invoke(
    f"""
请从下面招聘描述中提取结构化信息。
要求：
- 不要编造未提及的信息
- 技能请提取为列表
- 如果远程政策不明确，remote 返回 null

招聘描述：
{job_text}
"""
)

print("\n=== 结构化提取结果 ===\n")
print(result)

print("\n=== 程序后处理示例 ===\n")
if result.remote is True:
    print("远程政策：支持远程")
elif result.remote is False:
    print("远程政策：不支持远程")
else:
    print("远程政策：未明确")

if "LangChain" in result.skills or "RAG" in result.skills:
    print("岗位特点：包含 LLM 应用相关经验要求")
else:
    print("岗位特点：暂未发现明显 LLM 应用经验要求")
```

运行：

```powershell
uv run examples/lesson_04_job_extractor_cli.py
```

建议输入：

```text
云舟智能招聘高级 AI 应用工程师，地点深圳，要求熟悉 Python、LangChain、RAG、FastAPI，有向量数据库经验优先。薪资 30k-45k，可每周远程一天。
```

观察重点：

- 程序可以基于 `result.remote` 做条件判断。
- 程序可以基于 `result.skills` 做列表判断。
- 这就是结构化输出服务后续业务逻辑的方式。

## 十四、常见错误与排查

### 1. 字段描述太模糊

不太好：

```python
info: str = Field(description="info")
```

更好：

```python
salary_range: str | None = Field(
    default=None,
    description="The salary range if mentioned",
)
```

字段描述越清楚，模型越容易知道该填什么。

### 2. 把所有字段都设成字符串

不太好：

```python
skills: str
remote: str
```

更好：

```python
skills: list[str]
remote: bool | None = Field(default=None)
```

类型本身就是提示，也是程序校验规则。

### 3. 要求模型编造缺失字段

如果文本没提到公司名，就不要强迫模型输出公司名。

使用可选字段：

```python
company: str | None = Field(
    default=None,
    description="The company name if mentioned",
)
```

同时在 Prompt 中说明：

```text
不要编造未提及的信息。
```

### 4. 模型或服务商不支持结构化输出

并不是所有模型和兼容接口都同等支持结构化输出。

如果 `with_structured_output()` 报错，优先检查：

- 当前模型是否支持 tool calling 或 provider-native structured output。
- 当前服务商是否完整兼容 OpenAI 的工具调用或结构化输出能力。
- 是否需要更换模型或供应商集成包。
- 如果 `method="json_schema"` 不被兼容接口支持，可以尝试去掉该参数，使用默认 function calling 方式，也就是本课示例 2 的写法。
- 如果默认 function calling 方式也失败，说明该模型或接口可能不完整支持工具调用，需要更换模型、服务商或结构化输出策略。

LangChain 官方 OpenAI 集成文档明确标注 `ChatOpenAI` 支持 structured output，但第三方兼容接口的细节可能不同。

### 5. 结构化输出不是绝对可靠

结构化输出能显著提高可用性，但不代表模型永远正确。

仍然需要：

- 清晰 schema。
- 清晰 Prompt。
- 合理的可选字段。
- 业务侧校验。
- 必要时人工复核。

## 十五、本课小结

这一课你完成了从“文本输出”到“字段输出”的升级：

- 自由文本适合人读，不适合程序长期稳定消费。
- 结构化输出让模型按 schema 返回数据。
- Pydantic 用来描述字段、类型和字段说明。
- `Field(description=...)` 能帮助模型理解字段含义。
- `with_structured_output()` 会返回一个结构化模型调用对象。
- `method="json_schema"` 使用 OpenAI 原生 structured output 能力。
- 不传 `method` 时，`ChatOpenAI` 默认使用 function calling 方式，更适合部分第三方兼容接口。
- 调用结构化模型后，可以直接拿到 Pydantic 对象。
- 可选字段建议使用 `类型 | None` 加 `Field(default=None, ...)`，既表达允许为空，也表达字段可以缺省。
- 结构化输出是 RAG、Tools、Agent、LangGraph 决策的重要基础。

请记住：

> Prompt 让模型理解任务，Schema 让模型和程序对齐结果。
