## 二、为什么需要 Output Parser

聊天模型最自然的输出是文本，LangChain 的聊天模型通常返回的是 `AIMessage`。

但真实应用里，你经常并不只是想“打印一段话”。你可能需要：

- 把回答作为普通字符串显示在界面里。
- 从文本中提取关键词列表。
- 让模型返回 JSON，然后交给后端程序继续处理。
- 把结果转换成带字段约束的业务对象。
- 在链路中把一个模型的输出交给下一个步骤。

如果没有 Output Parser，你很容易把解析逻辑写散：

```python
message = model.invoke(prompt_value)
text = message.text
items = text.split(",")
```

这在小脚本里能跑，但在项目里会带来几个问题：

- 每个地方都自己解析，规则不统一。
- Prompt 里要求的格式和代码里解析的格式容易脱节。
- 模型输出稍微多一句解释，解析就可能失败。
- 链路组合时，下游不知道上游到底输出什么类型。

Output Parser 的设计意图，就是把“模型输出如何变成程序数据”这件事从业务代码里抽出来。

你可以把它理解成：

> Output Parser 是模型输出和业务程序之间的格式转换层。

它不负责让模型更聪明，而是负责让模型输出更可控、更容易进入程序流程。

---

## 三、Output Parser 和结构化输出的关系

第 4 课你已经学过 `with_structured_output()`。

这里要先明确一个原则：

> 如果模型和供应商支持原生结构化输出，并且你的目标是稳定得到业务对象，优先考虑 `with_structured_output()`。

那为什么还要学 Output Parser？

因为 Output Parser 仍然很常用：

- 你只需要把 `AIMessage` 转成字符串。
- 你要处理简单列表、简单 JSON 或自定义文本格式。
- 你使用的第三方 OpenAI-compatible 接口不完整支持结构化输出。
- 你希望在 LCEL 中显式表达“最后一步是格式转换”。
- 你需要兼容已有 Prompt，它已经通过文字要求模型输出某种格式。

可以这样理解两者的分工：

| 方式 | 更适合 |
| --- | --- |
| `with_structured_output()` | 让模型按 schema 生成结构化结果，适合稳定业务对象 |
| Output Parser | 对模型消息做解析和转换，适合文本、列表、JSON、兼容方案 |

它们不是互相替代的关系，而是不同层面的工具。

## 四、示例 1：`StrOutputParser`

最常见的解析器是 `StrOutputParser`。

它解决的问题很简单：模型返回的是 `AIMessage`，但你希望链路最终返回字符串。

新建文件：

```text
examples/lesson_07_str_output_parser.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_core.output_parsers import StrOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI


load_dotenv()


prompt = ChatPromptTemplate.from_messages(
    [
        ("system", "你是一名简洁的技术解释助手。"),
        ("human", "请用一句话解释：{topic}"),
    ]
)

model = ChatOpenAI(
    model=getenv("OPENAI_MODEL", "gpt-5-nano"),
    temperature=0.2,
)

parser = StrOutputParser()

chain = prompt | model | parser

result = chain.invoke({"topic": "Output Parser"})

print(type(result))
print(result)
```

运行：

```powershell
uv run examples/lesson_07_str_output_parser.py
```

观察输出：

- `result` 的类型是 `str`。
- 你拿到的是最终文本，而不是 `AIMessage`。

在大多数基础问答链、摘要链、改写链里，`StrOutputParser()` 都是最常见的最后一步。

## 五、什么是格式指令

有些解析器不仅会解析输出，还会告诉你应该怎样要求模型输出。

这类提示通常叫 **format instructions**，也就是格式指令。

它的作用是把解析器的期待格式写进 Prompt：

```text
请按下面格式输出：
...
```

为什么要这样设计？

因为 Parser 只能解析已经生成出来的文本，它不能凭空保证模型一定按格式写。

所以你通常要做两件事：

```text
Prompt 负责要求模型按格式输出
Parser 负责把这个格式解析成 Python 数据
```

这就是格式指令的价值：让 Prompt 和 Parser 使用同一套格式约定。

## 六、示例 2：关键词列表提取器

现在构建一个关键词列表提取器。

目标是输入一段文本，输出 Python 列表：

```python
["LCEL", "Runnable", "Output Parser"]
```

新建文件：

```text
examples/lesson_07_keyword_list_parser.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_core.output_parsers import CommaSeparatedListOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI


load_dotenv()


parser = CommaSeparatedListOutputParser()

prompt = ChatPromptTemplate.from_messages(
    [
        ("system", "你只负责提取关键词，不要解释。"),
        (
            "human",
            "请从下面文本中提取 5 个中文关键词。\n\n"
            "文本：{text}\n\n"
            "{format_instructions}",
        ),
    ]
)

model = ChatOpenAI(
    model=getenv("OPENAI_MODEL", "gpt-5-nano"),
    temperature=0,
)

chain = prompt | model | parser

result = chain.invoke(
    {
        "text": "LCEL 让 Prompt、Model 和 Parser 可以通过管道组合成可复用链路。",
        "format_instructions": parser.get_format_instructions(),
    }
)

print(type(result))
print(result)
```

运行：

```powershell
uv run examples/lesson_07_keyword_list_parser.py
```

请观察：

- 输出不再是字符串，而是 Python `list`。
- `parser.get_format_instructions()` 会提供解析器期待的输出格式说明。
- Prompt 负责让模型按格式输出，Parser 负责把输出转成列表。

这类解析器适合简单列表结果，例如关键词、标签、类别候选项。

## 七、示例 3：JSON 输出解析

很多业务系统喜欢 JSON，因为它可以自然表示字段、数组和嵌套结构。

下面做一个文章元信息提取器，输出字典。

新建文件：

```text
examples/lesson_07_json_output_parser.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_core.output_parsers import JsonOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI


load_dotenv()


parser = JsonOutputParser()

prompt = ChatPromptTemplate.from_messages(
    [
        (
            "system",
            "你是一个信息抽取助手。只输出 JSON，不要输出 Markdown 代码块。",
        ),
        (
            "human",
            "请从下面文本中提取文章信息。\n\n"
            "文本：{text}\n\n"
            "JSON 字段要求：\n"
            "- title: 适合这段文本的标题\n"
            "- summary: 一句话摘要\n"
            "- tags: 3 个关键词组成的数组\n\n"
            "{format_instructions}",
        ),
    ]
)

model = ChatOpenAI(
    model=getenv("OPENAI_MODEL", "gpt-5-nano"),
    temperature=0,
)

chain = prompt | model | parser

result = chain.invoke(
    {
        "text": "LangChain 的 Runnable 接口让 Prompt、模型、解析器等组件可以统一调用和组合。",
        "format_instructions": parser.get_format_instructions(),
    }

print(type(result))
print(result)
print(result["title"])
```

运行：

```powershell
uv run examples/lesson_07_json_output_parser.py
```

观察输出：

- `result` 通常是 Python 字典。
- 你可以用 `result["title"]` 访问字段。
- Prompt 里明确要求不要输出 Markdown 代码块，这能减少解析失败。

JSON 解析器适合你想得到字典结构，但还不需要严格 Pydantic 字段校验的场景。

## 八、在 Prompt 模板中写 JSON 示例要注意双花括号

如果你直接在 `ChatPromptTemplate` 字符串里写 JSON 示例，会遇到一个细节。

Prompt 模板使用 `{name}` 表示变量，所以普通 JSON 的花括号可能被误认为变量。

错误示例：

```python
prompt = ChatPromptTemplate.from_template(
    "请输出 JSON：{\"title\": \"...\", \"tags\": [\"...\"]}"
)
```

更稳妥的写法是使用双花括号转义：

```python
prompt = ChatPromptTemplate.from_template(
    "请输出 JSON：{{\"title\": \"...\", \"tags\": [\"...\"]}}"
)
```

含义是：

```text
{{ 和 }} 表示普通文本里的 { 和 }
{text} 才表示 Prompt 变量
```

这一点在 JSON 提示词中非常常见。你在第 4 课已经见过类似写法，这里再次强调，是因为 Output Parser 很依赖模型按格式输出。

## 九、示例 4：Pydantic 输出解析

如果你希望结果更像业务对象，可以使用 `PydanticOutputParser`。

注意：这不是在替代第 4 课的 `with_structured_output()`。

更推荐的理解是：

- `with_structured_output()`：让模型端按 schema 产出结构化结果。
- `PydanticOutputParser`：让模型先输出文本，再由解析器解析并校验成 Pydantic 对象。

新建文件：

```text
examples/lesson_07_pydantic_output_parser.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_core.output_parsers import PydanticOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI
from pydantic import BaseModel, Field


load_dotenv()


class ArticleInfo(BaseModel):
    title: str = Field(description="文章标题")
    summary: str = Field(description="一句话摘要")
    tags: list[str] = Field(description="关键词列表")


parser = PydanticOutputParser(pydantic_object=ArticleInfo)

prompt = ChatPromptTemplate.from_messages(
    [
        (
            "system",
            "你是一个信息抽取助手。只输出符合要求的 JSON，不要输出解释。",
        ),
        (
            "human",
            "请从下面文本中提取文章信息。\n\n"
            "文本：{text}\n\n"
            "{format_instructions}",
        ),
    ]
)

model = ChatOpenAI(
    model=getenv("OPENAI_MODEL", "gpt-5-nano"),
    temperature=0,
)

chain = prompt | model | parser

result = chain.invoke(
    {
        "text": "Output Parser 可以把模型生成的文本转换成字符串、列表、JSON 或业务对象。",
        "format_instructions": parser.get_format_instructions(),
    }
)

print(type(result))
print(result)
print(result.title)
```

运行：

```powershell
uv run examples/lesson_07_pydantic_output_parser.py
```

观察输出：

- `result` 是 `ArticleInfo` 对象。
- 你可以通过 `result.title` 访问字段。
- 如果模型输出不符合 schema，解析阶段可能报错。

在支持原生结构化输出的模型上，优先使用第 4 课的 `with_structured_output()`；在需要兼容文本格式解析、旧 Prompt 或第三方接口时，`PydanticOutputParser` 仍然有价值。

## 十、解析器在 LCEL 中的位置

常见链路是：

```python
chain = prompt | model | parser
```

为什么 Parser 通常放在最后？

因为它的职责是把模型输出变成程序使用的数据：

```text
PromptValue → AIMessage → str / list / dict / Pydantic object
```

如果你把 Parser 放在模型前面，就会出现类型不匹配：

```python
chain = prompt | parser | model
```

这通常是不合理的。Parser 期待模型输出，而不是 PromptValue。

学习 LCEL 时，请始终问自己：

```text
这一段的输入是什么？
这一段的输出是什么？
下一段能接住这个输出吗？
```

这个习惯比记住更多类名更重要。

## 十一、格式越复杂，越要降低模型自由度

Output Parser 的一个现实问题是：模型输出不是编译器输出。

如果你的 Prompt 很含糊：

```text
请提取一些信息。
```

模型可能输出解释、标题、Markdown、项目符号，甚至混合格式。

Parser 解析这种内容就会不稳定。

更好的写法是：

```text
只输出 JSON。
不要输出 Markdown 代码块。
不要添加解释。
字段必须包含 title、summary、tags。
```

这并不是为了“命令模型听话”这么简单，而是为了让生成端和解析端达成同一份格式契约。

可以把它理解为：

```text
Prompt 负责约束生成空间
Parser 负责验证和转换结果
```

格式越严格，Prompt 越要明确；解析越重要，越应该降低模型的自由发挥空间。

## 十二、实践案例：关键词列表提取器增强版

现在把本课内容做成一个更完整的小案例：从文本中提取关键词，并用于后续程序逻辑。

新建文件：

```text
examples/lesson_07_keyword_extractor_practice.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_core.output_parsers import CommaSeparatedListOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI


load_dotenv()


parser = CommaSeparatedListOutputParser()

prompt = ChatPromptTemplate.from_messages(
    [
        (
            "system",
            "你是一个关键词提取器。只输出关键词列表，不要解释。",
        ),
        (
            "human",
            "请从下面文本中提取 5 个适合作为搜索标签的关键词。\n\n"
            "文本：{text}\n\n"
            "{format_instructions}",
        ),
    ]
)

model = ChatOpenAI(
    model=getenv("OPENAI_MODEL", "gpt-5-nano"),
    temperature=0,
)

keyword_chain = prompt | model | parser

keywords = keyword_chain.invoke(
    {
        "text": "RAG 应用通常会先把文档切分、向量化并存入向量数据库，然后根据用户问题检索相关片段。",
        "format_instructions": parser.get_format_instructions(),
    }
)

print("关键词：")
for keyword in keywords:
    print("-", keyword)

print("\n可以交给后续程序使用的数据：")
print({"keywords": keywords, "count": len(keywords)})
```

运行：

```powershell
uv run examples/lesson_07_keyword_extractor_practice.py
```

请观察最后打印的字典：

```python
{"keywords": keywords, "count": len(keywords)}
```

这一步很重要。它说明 Output Parser 的目标不是让输出“看起来整齐”，而是让结果能继续被程序使用。

## 十三、常见错误与排查

### 1. Prompt 没有写格式要求

只写：

```text
请提取关键词。
```

模型可能返回一整段解释，列表解析器就可能失败。

解决方法：使用解析器的 `get_format_instructions()`，并在 Prompt 中明确“不需要解释”。

### 2. 要求 JSON，却输出了 Markdown 代码块

模型有时会输出：

~~~text
```json
{...}
```
~~~

这可能导致某些解析场景失败。

解决方法是在 Prompt 中写清楚：

```text
只输出 JSON，不要输出 Markdown 代码块。
```

### 3. 把 JSON 解析器当成严格 schema 校验器

`JsonOutputParser` 适合把 JSON 文本解析成 Python 字典。

如果你需要字段类型、必填项和业务约束，考虑：

- 使用 `with_structured_output()`。
- 或使用 `PydanticOutputParser`。

### 4. 忘记 Parser 解析的是模型输出

下面这种链路通常不对：

```python
chain = prompt | parser | model
```

Parser 一般放在模型后面：

```python
chain = prompt | model | parser
```

### 5. 温度太高导致格式不稳定

当你要求模型输出 JSON、列表或固定字段时，建议把 `temperature` 设置得低一些，例如：

```python
temperature=0
```

这不能保证永远不出错，但能减少自由发挥。

### 6. 解析失败时只看最终报错

排查解析失败时，先临时去掉 Parser，看模型原始输出：

```python
debug_chain = prompt | model | StrOutputParser()
print(debug_chain.invoke(input_data))
```

确认模型到底输出了什么，再决定是改 Prompt，还是换解析策略。

## 十四、本课小结

这一课你学习了链路中的最后一段：Output Parser。

你需要记住：

- 模型输出和程序需要的数据不是一回事。
- Output Parser 负责把模型输出转换成程序更容易使用的格式。
- `StrOutputParser` 常用于得到普通字符串。
- `CommaSeparatedListOutputParser` 可以把逗号分隔内容解析成列表。
- `JsonOutputParser` 可以把 JSON 文本解析成字典。
- `PydanticOutputParser` 可以把文本解析并校验成 Pydantic 对象。
- 解析器通常放在 `prompt | model | parser` 的最后。
- 格式越严格，Prompt 越要明确。

`with_structured_output()` 更适合原生结构化输出；Output Parser 更适合链路末端的格式转换、简单解析和兼容场景。
