# 第 9 课：可复用链路封装

> 本课关键词：**链路封装、业务函数、输入输出契约、目录结构、文本处理台、可维护性**

---

## 一、本课目标

学完本课，你应该能做到：

1. 理解为什么示例脚本需要进一步整理成可复用代码。
2. 使用 `build_xxx_chain(model)` 封装单一业务链。
3. 设计清晰的输入输出契约，减少字段传递错误。
4. 把多个文本处理能力组合成一个简单工作台。
5. 判断什么时候应该拆分目录，什么时候保持简单函数就够了。
6. 为后续项目实践建立基本的代码组织习惯。

到这里，你已经能把 Prompt、模型、解析器、透传和并行组合起来。下一步不是继续堆更多组件，而是学会把这些能力整理成“以后还能用、别人也能读懂”的代码。

---

## 二、从能跑到可复用

学习阶段的代码经常长这样：

```python
prompt = ...
model = ...
parser = ...
chain = prompt | model | parser
result = chain.invoke({"text": "..."})
print(result)
```

这很适合验证一个概念。但当你开始做一个真正的小工具时，问题会很快出现：

- 每个文件都重复创建模型。
- Prompt、Parser、业务调用混在一起。
- 同一个输入字段，有的地方叫 `text`，有的地方叫 `content`。
- 新增一个功能时，只能复制粘贴一段旧代码再改。
- 批量处理、命令行入口、Web API 入口无法共用同一条链。

可复用封装要解决的不是“代码看起来更高级”，而是让一个能力有稳定边界：

```text
输入是什么
输出是什么
内部怎么组合
外部怎么调用
后续怎么扩展
```

本课会把阶段二的文本处理能力整理成一个小型“文本处理台”。

---

## 三、先写输入输出契约

封装链路之前，先写清楚输入输出契约。

契约就是调用方和链路之间的约定。比如摘要链：

```text
输入：
{
    "text": str
}

输出：
str
```

再比如综合分析链：

```text
输入：
{
    "text": str
}

输出：
{
    "title": str,
    "summary": str,
    "keywords": list[str]
}
```

这一步看起来像文档，实际上是在帮你提前发现问题：

- Prompt 需要 `{text}`，调用方就不能传 `{article}`。
- 关键词链返回 `list[str]`，调用方就不应该把它当字符串拼接。
- 综合分析链返回字典，后续代码就应该通过 key 访问字段。

你写的链越多，输入输出契约越重要。

## 四、示例 1：封装摘要链

先从一个最小可复用链开始。

新建文件：

```text
examples/lesson_09_summary_chain.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_core.output_parsers import StrOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI


def create_model() -> ChatOpenAI:
    return ChatOpenAI(
        model=getenv("OPENAI_MODEL", "gpt-5-nano"),
        temperature=0.2,
    )


def build_summary_chain(model: ChatOpenAI):
    prompt = ChatPromptTemplate.from_messages(
        [
            ("system", "你是一名技术内容摘要助手。"),
            ("human", "请用一句话总结下面文本：\n\n{text}"),
        ]
    )

    return prompt | model | StrOutputParser()


def main() -> None:
    load_dotenv()

    model = create_model()
    summary_chain = build_summary_chain(model)

    result = summary_chain.invoke(
        {
            "text": "LangChain 的 Runnable 接口让 Prompt、模型和解析器可以用统一方式调用和组合。"
        }
    )

    print(result)


if __name__ == "__main__":
    main()
```

运行：

```powershell
uv run examples/lesson_09_summary_chain.py
```

这个例子里有三个层次：

- `create_model()` 负责模型创建。
- `build_summary_chain()` 负责摘要链构建。
- `main()` 负责准备输入、调用链、打印结果。

这就是最朴素、也最实用的封装方式。

## 五、为什么返回 chain

你也可以写一个直接返回摘要的函数：

```python
def summarize(text: str) -> str:
    return summary_chain.invoke({"text": text})
```

这在应用层可以存在，但构建层更推荐返回 chain：

```python
summary_chain = build_summary_chain(model)
```

因为返回 chain 以后，它仍然保留 Runnable 的统一能力：

```python
summary_chain.invoke({"text": "..."})
summary_chain.batch([{"text": "..."}, {"text": "..."}])
summary_chain.stream({"text": "..."})
```

也就是说，封装不是把 LangChain 的能力藏起来，而是把创建逻辑收拢，让调用方得到一个清晰的、可复用的 Runnable。

## 六、示例 2：封装标题、摘要和关键词链

现在把三个常见文本处理能力分别封装。

关键词链里会继续使用前面见过的两个小工具：

- `parser.get_format_instructions()`：从解析器拿到模型应该遵守的输出格式说明。
- `.partial(format_instructions=...)`：把这段格式说明提前固定到 Prompt 里，让调用方只需要传入 `text`。

新建文件：

```text
examples/lesson_09_text_chains.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_core.output_parsers import CommaSeparatedListOutputParser, StrOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI


def create_model() -> ChatOpenAI:
    return ChatOpenAI(
        model=getenv("OPENAI_MODEL", "gpt-5-nano"),
        temperature=0.2,
    )


def build_title_chain(model: ChatOpenAI):
    prompt = ChatPromptTemplate.from_messages(
        [
            ("system", "你是一名中文标题编辑。"),
            ("human", "请为下面文本生成一个不超过 18 个字的标题：\n\n{text}"),
        ]
    )

    return prompt | model | StrOutputParser()


def build_summary_chain(model: ChatOpenAI):
    prompt = ChatPromptTemplate.from_messages(
        [
            ("system", "你是一名技术内容摘要助手。"),
            ("human", "请用一句话总结下面文本：\n\n{text}"),
        ]
    )

    return prompt | model | StrOutputParser()


def build_keyword_chain(model: ChatOpenAI):
    parser = CommaSeparatedListOutputParser()

    prompt = ChatPromptTemplate.from_messages(
        [
            ("system", "你只负责提取关键词，不要解释。"),
            (
                "human",
                "请从下面文本中提取 5 个关键词。\n\n"
                "文本：{text}\n\n"
                "{format_instructions}",
            ),
        ]
    ).partial(format_instructions=parser.get_format_instructions())

    return prompt | model | parser


def main() -> None:
    load_dotenv()

    model = create_model()

    title_chain = build_title_chain(model)
    summary_chain = build_summary_chain(model)
    keyword_chain = build_keyword_chain(model)

    input_data = {
        "text": "LCEL 可以把 Prompt、Model、Parser 组合成可复用链路，RunnableParallel 可以组织多个分支的输出。"
    }

    print("标题：")
    print(title_chain.invoke(input_data))

    print("\n摘要：")
    print(summary_chain.invoke(input_data))

    print("\n关键词：")
    print(keyword_chain.invoke(input_data))


if __name__ == "__main__":
    main()
```

运行：

```powershell
uv run examples/lesson_09_text_chains.py
```

请观察：

- 三个链共享同一个 `model`。
- 每个 `build_xxx_chain()` 只负责一件事。
- 调用方统一传入 `{"text": ...}`。
- 关键词链内部使用列表解析器，调用方只关心最终拿到 `list[str]`。

这是封装带来的第一个收益：内部细节稳定收拢，外部调用方式保持一致。

## 七、给链路命名

链路多起来后，名字会直接影响可读性。

不推荐：

```python
chain1 = ...
chain2 = ...
my_chain = ...
```

更推荐：

```python
title_chain = ...
summary_chain = ...
keyword_chain = ...
text_analysis_chain = ...
```

构建函数也一样：

```python
def build_summary_chain(model):
    ...


def build_text_analysis_chain(model):
    ...
```

命名不只是风格问题。一个清楚的名字会告诉你：

- 这条链属于哪个业务能力。
- 输入大概是什么。
- 输出大概会被谁使用。
- 出问题时应该先检查哪一段。

后面接入 tracing 时，清晰名称也会让调用轨迹更容易读。

## 八、示例 3：封装综合分析链

接下来把标题、摘要、关键词组合成一个综合分析链。

新建文件：

```text
examples/lesson_09_text_analysis_chain.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_core.output_parsers import CommaSeparatedListOutputParser, StrOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnableParallel
from langchain_openai import ChatOpenAI


def create_model() -> ChatOpenAI:
    return ChatOpenAI(
        model=getenv("OPENAI_MODEL", "gpt-5-nano"),
        temperature=0.2,
    )


def build_title_chain(model: ChatOpenAI):
    prompt = ChatPromptTemplate.from_template(
        "请为下面文本生成一个不超过 18 个字的中文标题：\n\n{text}"
    )
    return prompt | model | StrOutputParser()


def build_summary_chain(model: ChatOpenAI):
    prompt = ChatPromptTemplate.from_template(
        "请用一句话总结下面文本：\n\n{text}"
    )
    return prompt | model | StrOutputParser()


def build_keyword_chain(model: ChatOpenAI):
    parser = CommaSeparatedListOutputParser()
    prompt = ChatPromptTemplate.from_template(
        "请从下面文本中提取 5 个关键词。\n\n"
        "文本：{text}\n\n"
        "{format_instructions}"
    ).partial(format_instructions=parser.get_format_instructions())

    return prompt | model | parser


def build_text_analysis_chain(model: ChatOpenAI):
    return RunnableParallel(
        title=build_title_chain(model),
        summary=build_summary_chain(model),
        keywords=build_keyword_chain(model),
    ).with_config({"run_name": "text_analysis_chain"})


def main() -> None:
    load_dotenv()

    model = create_model()
    text_analysis_chain = build_text_analysis_chain(model)

    result = text_analysis_chain.invoke(
        {
            "text": "可复用链路封装能把 Prompt、模型和解析器整理成稳定的业务能力，便于后续项目复用。"
        }
    )

    print(result)


if __name__ == "__main__":
    main()
```

运行：

```powershell
uv run examples/lesson_09_text_analysis_chain.py
```

这里的输出结构应该接近：

```python
{
    "title": "...",
    "summary": "...",
    "keywords": [...]
}
```

`with_config({"run_name": "text_analysis_chain"})` 不会改变模型结果，它只是给链路附加一个更清楚的运行名称。现在你只需要知道它有助于调试；后面学习 LangSmith 时，这类命名会更有价值。

## 九、阶段项目：多功能文本处理台

现在把多个链整理成一个小型工作台。

这个工作台先支持：

| 任务 | 输出 |
| --- | --- |
| `title` | 标题字符串 |
| `summary` | 摘要字符串 |
| `keywords` | 关键词列表 |
| `analysis` | 标题、摘要、关键词组成的字典 |

新建文件：

```text
examples/lesson_09_text_workbench.py
```

代码：

```python
from os import getenv

from dotenv import load_dotenv
from langchain_core.output_parsers import CommaSeparatedListOutputParser, StrOutputParser
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.runnables import RunnableParallel
from langchain_openai import ChatOpenAI


def create_model() -> ChatOpenAI:
    return ChatOpenAI(
        model=getenv("OPENAI_MODEL", "gpt-5-nano"),
        temperature=0.2,
    )


def build_title_chain(model: ChatOpenAI):
    prompt = ChatPromptTemplate.from_template(
        "请为下面文本生成一个不超过 18 个字的中文标题：\n\n{text}"
    )
    return prompt | model | StrOutputParser()


def build_summary_chain(model: ChatOpenAI):
    prompt = ChatPromptTemplate.from_template(
        "请用一句话总结下面文本：\n\n{text}"
    )
    return prompt | model | StrOutputParser()


def build_keyword_chain(model: ChatOpenAI):
    parser = CommaSeparatedListOutputParser()
    prompt = ChatPromptTemplate.from_template(
        "请从下面文本中提取 5 个关键词。\n\n"
        "文本：{text}\n\n"
        "{format_instructions}"
    ).partial(format_instructions=parser.get_format_instructions())

    return prompt | model | parser


def build_analysis_chain(model: ChatOpenAI):
    return RunnableParallel(
        title=build_title_chain(model),
        summary=build_summary_chain(model),
        keywords=build_keyword_chain(model),
    )


def build_text_workbench(model: ChatOpenAI):
    return {
        "title": build_title_chain(model),
        "summary": build_summary_chain(model),
        "keywords": build_keyword_chain(model),
        "analysis": build_analysis_chain(model),
    }


def run_workbench(workbench: dict, task: str, text: str):
    if task not in workbench:
        available_tasks = ", ".join(workbench.keys())
        raise ValueError(f"未知任务：{task}。可选任务：{available_tasks}")

    return workbench[task].invoke({"text": text})


def main() -> None:
    load_dotenv()

    model = create_model()
    workbench = build_text_workbench(model)

    text = "LangChain 的 Runnable 和 LCEL 让模型调用、Prompt 模板、输出解析器可以组合成可复用链路。"

    print("摘要：")
    print(run_workbench(workbench, "summary", text))

    print("\n完整分析：")
    print(run_workbench(workbench, "analysis", text))


if __name__ == "__main__":
    main()
```

运行：

```powershell
uv run examples/lesson_09_text_workbench.py
```

这个工作台的核心是：

```python
{
    "title": title_chain,
    "summary": summary_chain,
    "keywords": keyword_chain,
    "analysis": analysis_chain,
}
```

也就是把任务名称映射到可调用链路。这样新增功能时，你只需要新增一个 `build_xxx_chain()`，再把它注册到 `build_text_workbench()`。

## 十、什么时候拆分目录

单文件示例能帮助你看清全貌。但如果文件开始同时承担这些职责，就该考虑拆分：

- 读取配置。
- 创建模型。
- 定义 Prompt。
- 构建链。
- 选择任务。
- 处理命令行输入。
- 打印结果。

一个简单的拆分方案是：

```text
text_workbench/
  __init__.py
  config.py
  models.py
  chains.py
  workbench.py
  cli.py
```

职责可以这样安排：

| 文件 | 职责 |
| --- | --- |
| `config.py` | 读取环境变量、集中管理配置 |
| `models.py` | 创建模型客户端 |
| `chains.py` | 构建标题、摘要、关键词等链 |
| `workbench.py` | 组合任务映射，提供 `run_workbench()` |
| `cli.py` | 处理命令行输入和输出 |

不要为了“看起来像项目”过早拆分。更好的顺序是：

```text
先写清楚单文件版本
  ↓
发现职责开始混杂
  ↓
按配置、模型、链路、入口拆分
```

这样每一次拆分都有理由。

## 十一、封装时控制抽象程度

封装不是越复杂越好。

当前阶段，下面这种函数式封装已经足够：

```python
def build_summary_chain(model):
    ...


def build_title_chain(model):
    ...
```

不必急着写：

```python
class UniversalChainFactory:
    ...
```

只有当你真的遇到下面这些情况时，再考虑更强的抽象：

- 多个链有大量重复构建逻辑。
- Prompt 需要统一从文件或数据库加载。
- 要动态切换不同模型供应商。
- 需要为链路写单元测试、集成测试和部署脚本。
- 多人协作时需要更稳定的模块边界。

在此之前，清楚的函数、稳定的输入输出、统一的模型创建方式，比复杂架构更重要。

## 十二、常见错误与排查

### 1. 每条链都自己创建模型

不要在每个 `build_xxx_chain()` 里重新写：

```python
model = ChatOpenAI(...)
```

更好的做法是外部创建一次模型，再传给链路构建函数：

```python
model = create_model()
summary_chain = build_summary_chain(model)
title_chain = build_title_chain(model)
```

这样以后要切换模型、温度或兼容接口时，只需要改一个地方。

### 2. 输入字段不统一

如果摘要链用 `text`，标题链用 `content`，工作台就会越来越难用。

在文本处理台里，先统一使用：

```python
{"text": "..."}
```

需要更多字段时，再谨慎扩展输入契约。

### 3. 函数职责太多

一个函数同时创建模型、构建链、读取用户输入、打印结果，会很难复用。

可以按职责拆成：

```text
create_model()
build_xxx_chain()
run_workbench()
main()
```

每个函数做一类事情，代码会自然变清楚。

### 4. 输出结构不稳定

`analysis` 如果有时返回字符串，有时返回字典，调用方就无法稳定处理。

建议在代码旁边写清楚：

```text
analysis 输出：
{
    "title": str,
    "summary": str,
    "keywords": list[str]
}
```

### 5. 过早抽象

如果你只有三四条链，用函数就够了。

过早引入复杂类、配置框架或插件系统，会让学习成本高于收益。

### 6. 没有给任务失败留出口

工作台示例里如果任务不存在，会抛出：

```python
ValueError
```

这是一个最小错误处理。真实项目里你可以进一步返回友好的错误消息，或者在 Web API 中转换成对应的 HTTP 响应。

## 十三、本课小结

这一课把阶段二的链式能力整理成了可复用项目雏形。

你需要记住：

- 学习阶段重在看清组件，项目阶段重在整理边界。
- `build_xxx_chain(model)` 是当前最实用的链路封装方式。
- 函数返回 chain，可以保留 `invoke()`、`batch()`、`stream()` 等 Runnable 能力。
- 输入输出契约能减少字段传递和类型理解错误。
- 多个链应该共享统一的模型创建逻辑。
- 工作台可以用“任务名 → 链路”的映射来组织。
- 目录拆分应该跟随职责变化，而不是为了形式提前复杂化。

到这里，你已经完成了从单次模型调用到可复用文本处理链的过渡。

- 哪个文件负责命令行入口。
